Abstract. Extracting tables from documents is a crucial task in any
document intelligence pipeline. Recently, transformer-based models have
demonstrated that they can be used effectively for automatic table structure
extraction. However, most existing work on transformer-based approaches taking
HTML, LaTeX token sequences as input is not explicitly optimized for
in HTML, LaTeX which represent the structure of the table. Since the
table representation of the table into HTML, LaTeX has a significant impact on the
model's performance, we hypothesize that the encoding method will improve the
model's performance.
We propose a new, optimized table-tokenization language (OTL) with a
unified topology and operator semantics. The benefits of OTL are: (1) it
reduces the token sequence length by 75%, (2) it improves the
sequence length to half of HTML on average. Consequently, based
on our empirical results we show that by simply replacing the HTML input
with OTL models, we get significant performance improvement of 12-score and 14-point
reduction as compared to baseline methods.
Popular table structure datasets will be published in OTL format to the
ML community.

Keywords: Table structure recognition, Data Representation, Transformer, Optimization.

4.1  Language Definition

In Figure 5 we illustrate how the OTSL is defined. To ensure, the OTSL follows only a token that directly describe a tabular structure based on an atomic 2D

The OTSL vocabulary is comprised of the following tokens:

"[" cell - a new table cell that either has or does not have of content
"^" cell - split-stacking cell merging with the left neighbor cell to create a
"<" cell - split-stacking cell merging with the upper neighbor cell to create a
"|" cell - new cell to merge cells both left and upper neighbor cells
">" cell - row link, switch to the next row

A notable attribute of OTSL is that it has the capability of achieving lossless conversion to HTML.

4.2  Language Syntax

The OTSL representation follows these syntax rules:

1. Labeling cell rules. The left neighbour of a "[" cell must be either
   a "," or end of row.
2. Uplinking cell rules. The upper neighbour of a "[" cell must be either
   a "," or end of column.
   The left neighbour of an "^" cell must be either another "^" cell or a "[" cell,
   and the upper neighbour of an "^" cell must be either another "[" cell.
3. First row rule. Only "[" and "^" cells are allowed in the first row.
4. First column rule. Only "[" cells and "," are allowed in the first column.
5. Rectangular nature. Every table must have an equal number of tokens, terminated with "," token.

The application of these rules gives OTSL a set of unique properties. First of all, the OTSL enforces a strictly rectangular structure representation, where every row contains the same row is a commentary of row and column with uniform cells. As a consequence, all text and cell structure tokens can be validated straight during sequence generation by looking at the predicted token can be validated straight during sequence generation by looking at the previously predicted sequence. As such, OTSL can guarantee that every generated structure is a valid table.

These characteristics can be easily learned by sequence generator networks, as demonstrated earlier below. We find strong indications that this pattern
