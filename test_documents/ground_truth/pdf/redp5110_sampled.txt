1.1 Security fundamentals

To adequately secure information assets within the organization, there are two fundamental steps in securing
information assets that must be described.

First of most important is the definition of a company security policy. Without a
security policy, there is no definition of what are acceptable practices for using, accessing,
and transmitting data. This policy defines by who, when, where, and how the company's
assets can be accessed and used.

The monitoring and assessment of adherence to the security policy determines whether
the security strategy is working. Often, IBM security consultants are asked to help evaluate
how an organization is doing with respect to security. A number of in-depth

accessed. They provide audit determine the level of security within a security policy to
provide a good control framework. Those assessments also help you identify any significant
changes in the security settings that are captured.

The second fundamental in securing data assets is the use of security controls. If
organizations do not have a security control in place, they do not have a mechanism for policy
enforcement. A security control like a specific policy can be enforced to secure data in

1.2 Current state of IBM i security

Because of the extremely secure nature of IBM i, many clients rely on the default system
settings to protect their business data that is stored in DB2 for i. In most cases, this means no
data access controls are imposed inside the database. The default setting for the Criteria Create
system value is *UNRESTRICTED. This is a long-standing option.

Even more disturbing is that many IBM clients remain in this state, despite the news
headlines and the significant costs that are involved with databases being compromised. This
is a significant compliance challenge to implement issue security policies.
A lighter access restriction is required. If you really want to protect your company's most
valuable assets, which is the data.

Traditionally, IBM i applications have employed menu-based security to counteract this default
security posture. This limits all users access to the data. The theory is that, with menu-based
security, users cannot access anything they are not through the menu. However, this
approach is ineffective; even if the user profile is restricted from running interactive
commands. The reason is that in today's connected world there is a multitude of interfaces
like Office-based tools, web tools, Java utilities and others that can query a database and provide
no object-level controls; users of these newer interfaces have an open door to your data.

Many businesses are trying to limit data access to a need-to-know basis. The security goal
is to prevent data access by unauthorized users. This can be achieved by using a role-based
security model with data access at row and column levels. Currently, object-level access at the
database and table level is the primary method to prevent unauthorized access. This approach
provides an all-or-nothing solution. For example, object-level controls allow a manager to
access all employee data in the HR employees table. Most security policies, however, allow
only for the employees that they manage.

1.3.1 Existing row and column control

Some IBM i clients have tried augmenting the all-or-nothing object-level security with SQL
views or other application logic. The problem with application-based logic is easy to bypass with all of the different data access interfaces that
are provided by the IBM i operating system, such as Open Database Connectivity (ODBC).

Using SQL views to limit access to a subset of the data in a table also has its own set of
challenges. First, there is the complexity of managing all of the SQL view objects that are
used for securing data access. Second, using a view-based security solution can be difficult
on the amount of data rows and the number of users increases.

The DB2 RCAC feature provides a more fine-grained solution. RCAC provides fine-grained,
"ALL/ONLY access rule can directly access all of the data in the underlying DB2 table and easily
bypass the security controls that are built into SQL views.

2.2 Separation of duties

Separation of duties is a business compliance requirement that ensures compliance with industry regulations or organizational
requirements and simplifies the management of authorized users. Separation of duties also ensures that critical administrative
functions cannot be performed by a single user. This principle is established across individuals such that one user does not possess unlimited authority, such as with the "ALLOBJ" authority.

For example, let's assume that a database administrator assigned the duty to create users on IBM i to
Margaret. According to IBM i 7.2, to grant permission, that user will have to have a special authority of
*ALLOBJ to carry out the job. This result is that Margaret has too much authority and any user who
has *ALLOBJ authority may do anything. This requirement allowed an RCAC security officer to create
controls, such as *ALLOBJ, by using an RCAC function usage ID and permission rule that restricts
the ability. Similarly, the Create (CREATE) privilege that Theresa has in DB2 on IBM i allows
Theresa to access the data in the PAYROLL table. This grant allows Theresa through her job
description and role to access the data in the PAYROLL table. This grant is assigned through the
PAYROLL task function, which is the role of the RCAC administrator.

In IBM i 7.2, the CHGM DB_SECADM function usage grants authorities, revokes authorities,
changes permissions, or changes the primary group without having to be the object owner of the
table. This feature allows the DBA to grant RCAC permissions through a normal role model defined
in a table.

CHGM DB_SECADM function usage can be granted only by a user with *SECADM special
authority and can be given to a user or a group.

CHGM DB_SECADM is not responsible for administration RCAC, which restricts which rows
a user is allowed to access in a table and whether a user is allowed to see information in
certain columns in a table.

A non-RCAC administrator can have the CHGM DB_SECADM function usage or authority to
design ID, but depending on other data privileges. The result is that the RCAC administrator
can design a new structure the RCAC constructs, but cannot grant themselves unauthorized
access.

3.3 VERIFY_GROUP_FOR_USER function

The VERIFY_GROUP_FOR_USER function was added in IBM i 7.2. Although it is primarily
intended for system security management within the IBM i security framework, it is for
use with RCAC in security policies. The first parameter must be one of these three special registers: SESSION_USER, USER, or
CURRENT_USER. The second and subsequent parameters are user profile names. These values are
validated for their existence, which means that you can specify the names of user profiles that
are not yet created without receiving any kind of error.

If a special register value is in the list of user profiles or if it is a member of a group profile
included in the list, the function returns a long integer value of 1. Otherwise, it returns a value
of 0.

Here is an example of using the VERIFY_GROUP_FOR_USER function:

1. There are user profiles for MGR, JANE, JUDY, and TONY.
2. The user profile JANE specifies a group of MGR.
3. If a user connected to the server using user profile JANE, all of the following location
   invocations return a value of 1:

VERIFY_GROUP_FOR_USER (CURRENT_USER, 'MGR')
VERIFY_GROUP_FOR_USER (CURRENT_USER, 'JANE', 'MB', 'STEVE')

The following function invocation returns a value of 0:

VERIFY_GROUP_FOR_USER (CURRENT_USER, 'JUDY', 'TONY')
